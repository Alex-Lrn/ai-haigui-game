# AI 海龟汤游戏开发指令（给 AI 的工作手册）

## 1. 项目概述
本项目是一个 AI 驱动的海龟汤推理游戏网站，包含三个核心页面：
- `Home`：游戏大厅，展示故事列表并进入某一局
- `Game`：游戏对话页，展示汤面（surface）与聊天交互（提问 -> AI 回答）
- `Result`：揭晓页，展示汤底（bottom）与可选复盘对话

核心规则：
- AI 主持人必须只输出三种结果之一：`是` / `否` / `无关`
- 禁止解释、禁用提示性内容、禁止剧透
- `bottom` 仅在 `Result` 页面展示，避免在其它页面或前端请求中泄露

## 2. 开发规范（必须遵守）
- 使用 TypeScript，保证类型安全
- 使用函数式组件 + React Hooks
- 样式使用 Tailwind CSS（统一“游戏官网风”深蓝 + 青蓝霓虹 + 玻璃拟态）
- 代码尽量模块化：组件归属到 `src/components/`，页面归属到 `src/pages/`
- 保持代码简洁，优先实现核心功能；避免过度工程化
- 在做非确定性改动前先提出澄清问题

## 3. 代码风格要求
- 组件名使用 `PascalCase`（如 `GameCard`、`ChatBox`）
- 函数名使用 `camelCase`（如 `askAI`、`buildPrompt`）
- 常量命名使用 `UPPER_SNAKE_CASE`（如 `VALID_ANSWERS`）
- 类型/接口命名以 `T` 开头（如 `TStory`、`TMessage`、`TGameStatus`）
- 少量必要注释即可：只解释复杂逻辑或关键约束，不要写“显而易见的注释”

## 4. 设计与 UI 约束（游戏官网风）
1. 全局视觉
   - 背景：深蓝渐变（深蓝/靛蓝色系）
   - 强调色：青蓝霓虹（适度使用外发光/描边）
   - 容器：玻璃拟态（半透明 + `backdrop-blur` + 细边框）
2. 动效原则
   - 动效克制：优先 `淡入/上浮/轻微发光`
   - 不使用强干扰的闪烁、快速大幅跳动
3. 关键组件外观
   - `Home`：故事卡片玻璃质感，hover 轻微上浮并增强霓虹边框
   - `Game`：聊天气泡区分用户/AI；AI 回答气泡保持与规则一致的视觉风格
   - `Result`：揭晓页强调排版与仪式感（如淡入、分隔线、段落层次）
4. 响应式
   - 移动端必须可用：输入框不遮挡、按钮可点击、聊天区域可滚动

## 5. AI 对话与输出安全（最重要）
- AI 只根据汤底判断问题关系，不进行额外推理与补全剧情
- 输出必须严格属于集合：
  - `是`
  - `否`
  - `无关`
- 输出格式要求（建议强约束）：
  - 返回单个词或仅包含该词（不得包含标点/说明/换行的额外内容）
- 后端输出校验（必须做，若你负责实现后端）：
  - 对 AI 返回进行枚举校验
  - 校验失败：优先二次约束重试（最多 1 次）；仍失败则降级为 `无关`
- 前端兜底（若你负责前端）：
  - 若收到非三选一内容，以 system 消息引导用户换问法，并不展示异常原文

## 6. 测试要求（实现后要做）
- 每完成一个核心功能点（页面/接口/组件交互），执行以下手动检查：
  - 页面布局在桌面与移动端是否可读可用
  - 关键按钮能否正确触发路由/状态变化
  - AI 返回是否始终为 `是/否/无关`
  - 网络失败/超时是否展示友好错误提示且不破坏 UI
- 在提交实现前，确认没有暴露敏感信息：
  - API Key 绝对不能写入前端代码或公共文档
  - 只从环境变量读取（并在后端完成调用）

## 7. 交付方式建议
- 当你完成一个功能：给出“我改了哪些文件”和“我如何验证”的简短说明
- 当你遇到不确定点：先问我选择题（例如是否全量传 history、是否保留多轮消息等）
